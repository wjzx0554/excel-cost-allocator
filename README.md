# 分摊工具

一个本地 Windows 桌面工具，用于把 Excel/WPS 表格中的费用按指定基数列占比自动分摊。

## 主要功能

- 选择 `.xlsx` / `.xlsm` 文件。
- 选择工作表。
- 指定第几行是表头。
- 一个文件可配置多个分摊方案。
- 每个方案可单独设置方案名称、金额来源、分摊结果列、占比计算列和过滤条件。
- 金额来源支持：取分摊列原始合计、取指定金额列合计、手工输入金额。
- 过滤条件支持多个规则，规则关系支持 `OR` / `AND`。
- 过滤条件支持：等于、不等于、包含、不包含、正则、为空、非空。
- 执行前可先生成预览，核对金额、参与行数、不参与行数和基数合计。
- 输出新 Excel 文件，不修改原文件。
- 自动新增“分摊汇总”工作表，并为每个方案单独生成明细页，方便核对参与行、排除原因、占比和分摊结果。
- 支持保存 / 导入方案模板，下次处理同类表格时不需要重复配置列和过滤条件。
- 内置使用说明页、技术支持信息和测试数据生成按钮。

## 分摊规则

对每个分摊方案：

1. 确认待分摊金额来源。
2. 计算每行分摊基数 = 所选占比计算列的数值合计。
3. 过滤命中的行不参与分摊。
4. 分摊基数小于等于 0 的行不参与分摊。
5. 参与行金额 = 待分摊金额 × 本行分摊基数 / 参与行分摊基数合计。
6. 金额按 2 位小数四舍五入，尾差自动放到分摊基数最大的参与行。

## 推荐使用方式

财务实际场景里，建议“一笔费用对应一个分摊方案”，例如：

1. 共耗料分摊
2. 运费分摊
3. 水电费分摊
4. 手工金额临时分摊

这样每个方案单独核对，最稳，也最方便以后查账。

## 导出结果

批量方案导出后会新增：

- `分摊汇总`：每个方案一行，显示待分摊金额、参与行数、不参与行数、基数合计、分摊后合计和校验差额。
- `明细_01_方案名称`、`明细_02_方案名称`：每个方案一个独立明细页，显示源行号、是否参与、不参与原因、过滤命中说明、基数列、占比和分摊结果。

老版单方案接口仍保留 `分摊明细` 工作表，用于兼容已有逻辑。

## 方案模板

在“分摊方案”页可以：

- `保存模板`：把当前所有方案保存为 `.json` 文件。
- `导入模板`：读取 `.json` 模板，把方案应用到当前表格。

模板保存的是列名、方案、金额来源、占比列和过滤条件，不绑定某个月的 Excel 文件路径。导入时会按当前表头匹配列名；如果当前表缺少模板里的列，工具会提示缺少哪些列，避免静默分摊错误。

## 示例流程

1. 打开 Excel 文件。
2. 选择工作表。
3. 填写表头行并读取表头。
4. 到“分摊方案”页新增方案。
5. 选择分摊结果列。
6. 选择金额来源。
7. 多选占比计算列。
8. 添加过滤条件，例如“车间 等于 销售配货部”。
9. 常用方案可以点“保存模板”，以后读取新月份表头后点“导入模板”复用。
10. 先点“生成预览”。
11. 核对无误后点“开始分摊”。

## 测试数据样式

可以先用“生成测试数据”按钮创建一份示例表：

| 日期 | 车间 | 产品名称 | 产品类别 | 完工入库材料成本 wg_materialcost | 本期人工费 laborcost | 本期共耗料 consumptioncost | 运费分摊 |
|---|---|---|---|---:|---:|---:|---:|
| 2026-06 | 一车间 | A产品 | 成品 | 10000 | 3000 | 0 | 0 |
| 2026-06 | 二车间 | B产品 | 成品 | 20000 | 5000 | 0 | 0 |
| 2026-06 | 销售配货部 | 配货费用 | 内部 | 5000 | 1000 | 0 | 0 |
| 2026-06 | 三车间 | C产品 | 半成品 | 15000 | 4000 | 0 | 0 |

推荐测试方案：

- 方案名称：共耗料分摊
- 金额来源：手工输入金额 `100000`
- 占比计算列：完工入库材料成本 `wg_materialcost`、本期人工费 `laborcost`
- 分摊结果列：本期共耗料 `consumptioncost`
- 过滤条件：车间 等于 销售配货部

## 正则过滤示例

- `^销售配货部$`：只匹配销售配货部。
- `.*维修.*`：包含“维修”。
- `^(销售配货部|售后服务部)$`：匹配两个部门。
- `^$`：匹配空白内容。

## GitHub 自动打包

把整个项目上传到 GitHub 后，GitHub Actions 会在每次 push 或手动运行时，用 `flet_app.py` 自动生成新版 Flet UI 的 Windows x64 exe。

普通 push 或手动运行时，GitHub Actions 的 `Artifacts` 仍会以 zip 形式保存构建产物，这是 GitHub Actions 的默认机制。

如果希望用户直接下载 `.exe`，请打一个版本标签发布 Release：

```powershell
git tag v0.3.0
git push origin v0.3.0
```

发布后下载位置：

1. 打开 GitHub 仓库。
2. 进入 `Releases`。
3. 打开对应版本。
4. 在 `Assets` 里直接下载：
   - `FanchanTool-win-x64.exe`：64 位 Flet UI 版本，适合数据量较大的电脑。

## 本地开发运行

```powershell
python -m venv .venv
.venv\Scripts\activate
python -m pip install -r requirements-dev.txt
python flet_app.py
```

本地打包新版 UI：

```powershell
flet pack flet_app.py --yes --name FanchanTool --icon assets\app.ico --add-data "assets;assets"
```

生成的单文件程序位于 `dist\FanchanTool.exe`。旧版 Tkinter 入口 `main.py` 仍保留，但 GitHub Actions 默认打包的是 `flet_app.py`。

## 运行测试

```powershell
python -m pytest
```

## Windows 兼容说明

新版界面固定使用 Python 3.8、Flet 0.25.2 和 PyInstaller 5.13.2 构建。Flet 桌面程序包含 Flutter 窗口运行时，建议在 64 位 Windows 10 / 11 使用。

Flet 0.25.2 在 PyPI 上虽然存在名为 `win32` 的轮子，但其中的桌面运行时实际仍是 AMD64，不能作为可运行的 32 位程序发布，因此自动构建只生成 x64 新版 UI。

若必须支持 Windows 7 或 32 位 Windows，请继续使用旧版 Tkinter 入口 `main.py` 并单独构建和验收；GitHub Actions 默认发布的 Flet 新版 UI 不支持这些环境。
